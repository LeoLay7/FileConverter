Кейс «Конвертер форматов данных с гибкой настройкой правил» – направлен на проектирование и разработку инструмента (ETL-процессора) для трансформации структур данных между различными форматами с возможностью пользовательской настройки логики сопоставления.

# Постановка задания

## Название кейса: Конвертер форматов данных с гибкой настройкой правил (MVP).

## Цель выполнения кейса:
Развитие навыков разработки сложного программного обеспечения, работающего с метаданными и динамическими структурами. Студенты должны реализовать полный цикл обработки данных: от чтения исходного файла до генерации результирующего файла по заданным правилам.

## Конкретный образ результата:
1. Рабочий MVP веб-сервиса, развернутый на публичном хостинге (или доступный локально в контейнерах), который позволяет:
2. Загружать исходный файл данных (поддержка минимум двух форматов: JSON, XML, CSV).
3. Визуализировать структуру загруженного файла (дерево полей).
4. Настраивать правила конвертации (сопоставление полей, простейшие преобразования типов, переименование).
5. Получать и скачивать результирующий файл в целевом формате.

# Примерный план работы по спринтам (3 недели на спринт):

## Спринт 1 (Недели 1-3): Исследование и Архитектура
- Задачи: Выбрать форматы (например, JSON <-> XML). Спроектировать внутреннее представление данных (универсальную модель), в которую будут преобразовываться все входные форматы. Разработать формат хранения правил конвертации (конфигурационный файл или JSON-схема).
- Результат для Показа 1: Схема архитектуры, спецификация формата правил, Wireframes интерфейса.


## Спринт 2 (Недели 4-6): Парсеры и Сериализаторы
- Задачи: Реализовать чтение входных файлов в универсальную внутреннюю структуру. Реализовать выгрузку из внутренней структуры в целевые форматы. Обработка ошибок чтения файлов.
- Результат для Показа 2: Демонстрация: загружаем JSON -> получаем его структуру в консоли/логах -> сохраняем обратно в JSON (тождественное преобразование). То же самое для второго формата.


## Спринт 3 (Недели 7-9): Движок трансформации
- Задачи: Реализовать логику применения правил: переименование полей, удаление полей, изменение вложенности, простая типизация (строка -> число). Разработка API для выполнения конвертации.
- Результат для Показа 3: Демонстрация работы через Postman/Swagger: отправляем файл и конфиг правил -> получаем преобразованный файл. Интерфейса еще нет.


## Спринт 4 (Недели 10-12): Визуальный редактор (UI)
- Задачи: Разработка интерфейса загрузки файлов. Визуализация деревьев исходной и целевой схемы. Реализация механизма связывания полей (mapping) через UI. Интеграция фронтенда с бэкенд-API.
- Результат для Показа 4: Демонстрация MVP: пользователь загружает файл, видит его поля, выбирает, какие поля попадут в результат, и скачивает итог.


## Спринт 5 (Недели 13-15): Стабилизация и Финализация
- Задачи: Тестирование на сложных вложенных структурах. Оптимизация производительности. Улучшение UX (валидация, подсказки). Подготовка документации.
- Результат: Готовый проект, презентация.
