# Описание программной системы «Конвертер форматов данных с гибкой настройкой правил»

---

## 1. Введение и бизнес-контекст

Организации регулярно сталкиваются с задачей **переноса данных между информационными системами**, использующими разные форматы (JSON, XML, CSV и др.), при **несовпадении структуры полей**. Ручные преобразования и единоразовые скрипты слабо переносимы между проектами. Востребованы решения класса **No-Code/Low-Code**, позволяющие описывать логику конвертации настройкой правил без постоянной доработки кода интеграции.

Целевое **MVP** — веб-сервис, который обеспечивает: загрузку исходного файла поддерживаемых форматов, **визуализацию структуры данных в виде дерева полей**, **конфигурирование правил преобразования** и **получение результирующего файла** в целевом формате. По учебному заданию к решению предъявляются требования к освоению работы со **структурами данных в виде деревьев**, принципов **ETL**, безопасной обработке файлов и связке пользовательского интерфейса с программными интерфейсами сервера.

Форматы, на которые ориентируется решение: **JSON**, **XML**, **CSV** — с внутренним представлением на основе дерева данных.

---

## 2. Назначение системы для пользователя

### 2.1. Способы доступа

**Веб-интерфейс**

1. загрузка исходного файла;  
2. выполнение конвертации в выбранный формат по заданным настройкам;  
3. получение результирующего файла для скачивания.

**Программный интерфейс (API)**

1. работа через **идентификатор файла:** загрузка файла → получение идентификатора → запрос на конвертацию → получение ссылки на выгрузку → скачивание результата;  
2. работа **по содержимому в запросе:** передача текста исходных данных и целевого формата; в ответ — текст результата **или** ссылка на файл с результатом.

### 2.2. Соответствие требованиям учебного кейса

К MVP относятся: поддержка не менее двух форматов на входе и выходе, отображение дерева полей, настройка правил (сопоставление и типовые преобразования, в том числе переименование), выгрузка результата. Итоговая сдача проекта предполагает развёрнутый сервис или контейнеризованный вариант, репозиторий, техническую документацию и защиту. В календаре учебного кейса заданы **промежуточные контрольные точки:** архитектура и выбор стека → парсеры и сериализация → движок трансформации → связка UI и backend.

## 3. Предметная область: правила и настройки конвертации

### 3.1. Понятие правила

**Правило** — элементарная **операция преобразования** над структурой данных. Пример: замена типа значения по пути `content.views` с `string` на `int`.

Поля в иерархии адресуются **путём**; **уровни пути разделяются точкой.** Пример: преобразование поля `date` в объекте по пути `content.article1`.

### 3.2. Расширение логики (low-code)

Допускается определение **пользовательских правил** на языке **JavaScript** в сочетании с предопределёнными типами операций.

### 3.3. Базовый набор операций (MVP)

Ниже операции формулируются в терминах **внутренней древовидной модели** (после разбора файла JSON, XML или CSV). Чтобы однозначно интерпретировать правила, зафиксируем несколько понятий.

**Поле** — **именованная характеристика узла** в дереве: одна пара **«имя — значение»** по отношению к родительскому узлу (это не весь файл и не произвольный «набор данных» вообще, а **одно** именованное место во вложенной структуре). В JSON это ключ объекта; в XML — обычно **элемент** или **атрибут** в логической схеме, отображаемой как узел; в CSV, приводимой к дереву, поля задаются как **имена столбцов** (или согласованные с мэппингом обозначения) на нужном уровне. **Значение** поля может быть примитивом (строка, число, логический и т.д.), вложенным **объектом** (узел с дочерними полями) или **списком** (массив однотипных или разнотипных элементов).

**Путь к полю** — цепочка имён от корня документа к нужному полю через **точку** (например, `root.item.price` означает: в объекте `root`, внутри `item`, поле `price`). Операция правила всегда применяется к **одному адресу** в дереве — к конкретному полю или к каждому элементу списка согласно типу операции ниже.

**Объект (узел с полями)** — узел дерева, у которого есть именованный набор дочерних полей.

**Список объектов** — массив, все элементы которого трактуются как объекты с полями; для «пакетных» операций требуется, чтобы нужное имя поля было у **каждого** элемента списка (иначе сценарий считается некорректным для данной операции).

**Список значений** — массив **примитивов** (строки, числа и т.д. без вложенной структуры объекта правила доступа через дочерние поля по отдельности).

---

**Группа «Удаление»**

- **Удалить поле** по пути — у выбранного узла удаляется одно указанное **именованное** свойство вместе с его значением.  
- **Удалить поле (список объектов)** — для одной и той же коллекции объектов указанное подполе удаляется **в каждом** элементе списка (при выполнении условия наличия поля у всех элементов).

**Группа «Преобразование»**

- **Преобразовать тип** — значение конкретного поля приводится к целевому типу данных (например, строка «123» → число 123 при допустимой семантике).  
- **Преобразовать тип (список объектов)** — у каждого объекта в указанном списке тип одноимённого поля или поля по пути приводится к заданному типу.  
- **Преобразовать тип (список)** — каждый **элемент** одномерного списка примитивов приводится к заданному типу (не путать с предыдущим пунктом: там меняются однотипные поля внутри объектов массива).

**Добавление поля** — подлежит детальной проработке в следующих версиях описания функциональности.

### 3.4. Настройки конвертации

Правила объединяются в **настройки конвертации**, которые могут включать **простые и составные** правила. **Порядок выполнения** правил задаётся явно. Дополнительно допускается **выбор настройки конвертации сценарием** (скрипт) для конкретного файла или потока обработки.

## 4. Логическая архитектура системы

Система состоит из логических компонентов, приведённых в таблице. Назначение и границы ответственности согласованы с архитектурной моделью проекта и не изменяют состав решения.

| Компонент | Назначение |
|-----------|-------------|
| **Веб-клиент и API-шлюз** | Интерфейс пользователя; единая точка входа HTTP: маршрутизация, политики на периметре. |
| **User Service** | Аутентификация, авторизация, данные пользователей. |
| **Rule Service** | Конфигурация, хранение и жизненный цикл **правил** и связанных сущностей, включая **настройки конвертации**. |
| **Tree Transformer** | Выполнение конвертации для **JSON, XML и CSV**; внутренняя модель — **дерево** с одним корневым узлом и дочерними элементами. |
| **API Facade** | Синхронный приём операций управления задачами; **инициирование** асинхронной обработки через **Kafka**; хранение и выдача **состояния задачи**. Заложена возможность подключения **дополнительных типов трансформеров** без перегрузки фасада алгоритмами конкретной конвертации. |
| **File Service** | Загрузка файлов пользователей в **объектное хранилище** уровня S3 (**MinIO** в целевой инфраструктуре); управление **доступом к выгрузке** (например, предподписанные ссылки). |
| **Kafka** | Асинхронная доставка сообщений «задача исполнителю» и обратная передача событий о завершении. |
| **Rule DB**, **Facade DB** (PostgreSQL) | Разнесённые базы данных: данные конфигурации правил отдельно от данных задач и статусов. |
| **Redis** | **Кэш** для ускорения типовых чтений (например, конфигурации правил, данных о статусах задач); состав ключей и сроки актуальности задаются при детальном проектировании. |

**Потоки взаимодействия (обобщённо):** клиент обращается к шлюзу; шлюз — к прикладным сервисам; **Facade** обменивается сообщениями с **Kafka** и с **Rule Service**; **Tree Transformer** потребляет задания из очереди, обращается к **File Service** для чтения входного содержимого и записи результата.

## 5. Типовой сквозной сценарий (API и файл по идентификатору)

Сценарий применим к построению пользовательских историй и тест-кейсов.

1. Клиент выполняет **загрузку файла** через **File Service** и получает **идентификатор файла**.  
2. Клиент инициирует **конвертацию**, обращаясь к **Facade** с указанием идентификатора файла, целевого представления и **потока (flow)** / набора правил (точное оформление URI и тела запроса фиксируется в спецификации API, ориентир: `/transform/<file_id>/…/flow/<имя_потока>`). В ответ — **идентификатор задачи**.  
3. До перехода задачи в конечное состояние клиент **опрашивает Facade**; **активное состояние задачи учитывается на стороне Facade.**  
4. После успешного завершения Facade сообщает параметры **выгрузки**; клиент получает файл через **File Service** по выданному механизму доступа.

Сценарии работы только через браузер и сценарий «тело запроса = текст файла» реализуются в рамках перечисленных в разделе 2.1 способов доступа; детали контрактов API выносятся в отдельный документ.

---

## 6. Принятые архитектурные решения

- **Разделение баз данных** сервисов правил и фасада задач: снижение связности, независимое развитие и сопровождение конфигурации и операционных данных.  
- **Учёт статуса задачи** сосредоточен в **Facade**; клиентский опрос и внутренняя логика согласованы с этим источником данных.  
- **Асинхронное выполнение** конвертации через **Kafka** для выноса длительной обработки из синхронного запроса и для расширения линейки исполнителей.

---

## 7. Заключение

Описанная система представляет собой **веб-ориентированный ETL-инструмент** с **настраиваемыми правилами**, **двумя каналами потребления** (интерфейс и API) и **древовидной** моделью данных для целевых форматов. Длительная конвертация выполняется **асинхронно** (Facade, Kafka, Tree Transformer); **файлы** обслуживаются отдельным компонентом и **объектным хранилищем**.
