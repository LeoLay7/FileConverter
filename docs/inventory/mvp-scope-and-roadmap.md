# MVP-срез и план на 3 дня

## Цель MVP

Показать полный цикл ETL: файл загружен, структура распознана как дерево, пользователь настроил правила, система применила их к дереву и отдала результат в выбранном формате.

## Приоритеты

### P0 - обязательно

- Backend endpoint для загрузки файла.
- Автоопределение или ручной выбор формата: JSON, XML, CSV.
- Получение дерева полей из `JsonNode`.
- Движок правил над `JsonNode`.
- Конвертация результата через `tree-converter-lib`.
- Скачивание результата.
- Минимальный Web UI для upload/tree/rules/convert/download.

### P1 - желательно

- Swagger UI.
- Превью результата в UI.
- Сохранение истории задач в памяти.
- Dockerfile и `docker-compose.yml`.
- Импорт/экспорт JSON-набора правил.

### P2 - после MVP

- Kafka.
- PostgreSQL.
- MinIO.
- Redis.
- User Service.
- JavaScript custom rules.
- Drag-and-drop mapping.

## День 1 - Backend skeleton и дерево

Цель дня: файл можно загрузить и увидеть дерево.

Задачи:

- создать/дособрать Spring Boot приложение для MVP backend;
- подключить `tree-converter-lib`;
- реализовать `FileStoragePort` с локальной реализацией;
- реализовать `POST /api/files`;
- реализовать `GET /api/files/{fileId}/tree`;
- реализовать `TreeSchemaBuilder` для `JsonNode`;
- добавить обработку ошибок парсинга;
- покрыть тестом построение дерева.

Результат дня:

- curl/Postman: upload -> fileId -> tree;
- UI может быть еще статичным, но backend уже работает.

## День 2 - Правила и конвертация

Цель дня: файл преобразуется по правилам.

Задачи:

- реализовать модели `RuleSet`, `Rule`, `RuleType`, `TargetType`;
- реализовать `RuleEngine`;
- операции P0: `include/copy`, `rename`, `delete`, `typeCast`;
- реализовать `POST /api/transform`;
- сохранять результат в `FileStoragePort`;
- реализовать `GET /api/files/{fileId}/download`;
- добавить `JobStatus`, даже если обработка синхронная;
- покрыть тестами применение правил.

Результат дня:

- curl/Postman: upload -> transform -> download;
- есть 1-2 готовых rule-set JSON для демонстрации.

## День 3 - UI, упаковка, стабилизация

Цель дня: демонстрационный сценарий выполняется из браузера.

Задачи:

- реализовать экран загрузки файла;
- вывести дерево полей;
- добавить форму правила;
- добавить список правил с редактированием/удалением;
- добавить выбор target format;
- добавить кнопку Convert и ссылку Download;
- добавить preview результата, если хватает времени;
- подготовить README запуска;
- подготовить Docker Compose или простой локальный запуск;
- прогнать демо-сценарии JSON->CSV, CSV->JSON, XML->JSON.

Результат дня:

- MVP можно показать с экрана;
- есть короткая инструкция и набор тестовых файлов.

## Контрольные точки

- T+4 часа: backend стартует, Swagger/health доступен.
- T+8 часов: upload и tree работают.
- T+16 часов: rule engine применяет правила в тестах.
- T+24 часа: transform/download работают через API.
- T+32 часа: UI выполняет главный сценарий.
- T+40 часов: стабилизация, демо-данные, README.

## Главный компромисс

Полную микросервисную схему не реализовывать физически за 3 дня. Вместо этого реализовать границы через интерфейсы:

- `FileStoragePort` вместо отдельного File Service + MinIO;
- `RuleRepositoryPort` вместо Rule Service + PostgreSQL;
- `JobRepositoryPort` вместо Facade DB;
- `TransformExecutor` вместо Kafka + Tree Transformer.

После защиты эти порты можно заменить реальными сервисами.
