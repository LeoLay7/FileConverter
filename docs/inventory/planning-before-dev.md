# Что распланировать до начала разработки

Перед стартом нужно зафиксировать решения, которые иначе будут тормозить разработку и плодить несовместимые контракты между UI, backend и библиотекой.

## 1. Срез MVP

Нужно явно отделить обязательное от целевого.

Обязательно за 3 дня:

- загрузка файла через UI;
- поддержка JSON, XML, CSV минимум в одной рабочей трассе;
- построение дерева полей;
- настройка правил через UI или JSON-редактор правил;
- применение правил к `JsonNode`;
- сериализация результата в выбранный формат;
- скачивание результата;
- Swagger/OpenAPI или README с примерами запросов;
- Docker Compose или локальная инструкция запуска.

Можно отложить:

- Redis;
- JavaScript-скрипты правил;
- drag-and-drop mapping между двумя деревьями, если не успеваем.

## 2. Архитектурный режим MVP

Целевая архитектура описывает несколько сервисов. На 3 дня лучше выбрать один из двух режимов:

**Режим A - быстрый монолит MVP.** Один Spring Boot backend реализует upload, schema, rules, transform, download. Пакеты и интерфейсы именуются так, чтобы потом вынести их в File Service, Rule Service и Tree Transformer.

**Режим B - минимальные сервисы.** Отдельно `file-service`, `rule-service`, `facade-service`, но без Kafka/Redis/PostgreSQL. Риск выше: много времени уйдет на инфраструктуру.

Рекомендация: режим A. Для защиты можно показать, какие классы соответствуют будущим сервисам.

## 3. Формат правил

Нужно заранее зафиксировать JSON-формат правил. Минимальный набор:

- `include` - перенести поле из `sourcePath` в `targetPath`;
- `rename` - переименовать поле;
- `delete` - удалить поле;
- `typeCast` - привести значение к `string`, `integer`, `decimal`, `boolean`;
- `copy` - скопировать поле без изменения;
- `constant` - добавить поле с константой, если останется время.

Важно: правила выполняются строго по `order`.

## 4. Адресация полей

В документации указаны пути через точку. Для реализации на Jackson безопаснее использовать внутренний адаптер:

- внешний формат: `root.items.price`;
- внутреннее преобразование: список сегментов `["root", "items", "price"]`;
- массивы в MVP трактовать пакетно: если путь указывает на поле внутри массива объектов, операция применяется к каждому объекту массива.

На 3 дня не вводить сложный синтаксис индексов вроде `items[0]`, если он не нужен для демонстрации.

## 5. Дерево полей для UI

Нужно определить единый DTO дерева:

- `path`;
- `name`;
- `type`: `object`, `array`, `string`, `number`, `boolean`, `null`;
- `children`;
- `sampleValue`;
- `arrayItemType`, если узел массив.

Это позволит UI не знать про Jackson.

## 6. API

До разработки зафиксировать минимальные эндпоинты:

- `POST /api/files` - upload;
- `GET /api/files/{fileId}/tree` - дерево;
- `POST /api/transform` - конвертация;
- `GET /api/jobs/{jobId}` - статус, даже если внутри всё синхронно;
- `GET /api/files/{fileId}/download` - скачивание;
- `POST /api/preview` - опциональное превью результата без сохранения.

## 7. UI-сценарий

Нужно заранее нарисовать простой экран:

- верхняя панель: upload, source format, target format;
- левая область: дерево полей;
- правая область: список правил;
- нижняя/правая область: preview, convert, download.

Для скорости допустим гибрид: выбор поля из дерева + форма правила + JSON-preview правил.

## 8. Тестовые данные и демо-сценарий

До кода подготовить 3 файла:

- `users.csv` с колонками `id,name,age,active`;
- `catalog.xml` с вложенными элементами;
- `order.json` с объектом, массивом позиций и числовыми строками.

Демо должно быть коротким:

1. загрузить `users.csv`;
2. увидеть дерево колонок;
3. переименовать `name` в `fullName`;
4. привести `age` к integer;
5. удалить `active`;
6. скачать JSON.

## 9. Критерии приемки

MVP готов, если:

- приложение запускается одной командой;
- UI позволяет выполнить главный сценарий без Postman;
- backend имеет Swagger или документированные curl-примеры;
- минимум 3 автотеста покрывают парсинг дерева, применение правил, конвертацию;
- ошибки парсинга и неверных правил возвращаются понятным JSON.
