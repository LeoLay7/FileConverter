# Навигация по инвентарю MVP

Инвентарь фиксирует план разработки MVP сервиса "Конвертер форматов данных с гибкой настройкой правил" на 3 дня. Полная архитектура в `docs/arch.md` остается целевой, но для MVP используется суженный путь: быстро показать полный пользовательский сценарий от загрузки файла до скачивания результата.

## Главная MVP-трасса

1. Пользователь открывает Web UI.
2. Загружает JSON, XML или CSV.
3. Видит дерево полей.
4. Выбирает целевой формат.
5. Создает набор правил: include/map/rename/type/delete.
6. Запускает конвертацию.
7. Получает превью результата и скачивает файл.

## Рекомендуемый технический срез на 3 дня

Для успевания за 3 дня не поднимать всю микросервисную инфраструктуру сразу. Рекомендуется:

- оставить `tree-converter-lib` как ядро парсинга и сериализации;
- реализовать MVP backend в одном Spring Boot приложении или в `facade-service`, подключив библиотеку;
- хранение файлов сделать локальным на диске или in-memory, с интерфейсом, совместимым с будущим File Service;
- правила передавать в запросе и опционально сохранять в памяти/файле, без PostgreSQL на первом проходе;
- Kafka, Redis, полноценный User Service и отдельный API Gateway заменить прямыми HTTP-вызовами в MVP, но описать их как целевую эволюцию.
