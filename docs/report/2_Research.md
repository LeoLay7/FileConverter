# 2. Исследование

## 2.1. Логика работы системы

Разрабатываемая система предназначена для файловой конвертации с возможностью применения пользовательских правил преобразования. Общий сценарий включает загрузку исходного файла, сохранение сведений о нем, выбор целевого формата, настройку или выбор набора правил, запуск задания на обработку и получение результата.

После загрузки файл регистрируется в системе как отдельный объект. Его содержимое хранится отдельно от служебной информации, а метаданные используются для поиска, скачивания, удаления и запуска преобразования. Такой подход позволяет управлять файлами без смешивания бинарных данных и структурированных сведений.

Правила преобразования задаются независимо от файлов и могут переиспользоваться в разных заданиях. Набор правил описывает операции над структурой данных: перенос значения, переименование поля, удаление поля, изменение типа и обработку списков. Поддержка списков необходима для случаев, когда однотипное преобразование требуется применить к каждому элементу массива.

Для запуска конвертации клиент передает идентификатор исходного файла, целевой формат и, при необходимости, идентификатор набора правил. На основании этих данных создается задание, которому присваивается собственный идентификатор и начальный статус. Дальнейшая обработка выполняется асинхронно, поэтому пользовательский запрос не блокируется на время конвертации.

В процессе обработки файл приводится к общему древовидному представлению. Если для задания указаны правила, они применяются к этой промежуточной структуре. После этого результат сериализуется в целевой формат, сохраняется как новый файл, а состояние задания обновляется. При успешном завершении клиент получает сведения о результирующем файле, при ошибке сохраняется описание причины неуспешного выполнения.

Таким образом, логика системы основана на жизненном цикле задания: загрузка исходных данных, выбор параметров, запуск фоновой обработки, проверка статуса и получение результата.

## 2.2. Архитектура и основные архитектурные решения

Архитектура MVP построена по принципу разделения ответственности между несколькими backend-компонентами. Каждый компонент отвечает за отдельную функциональную область: работу с файлами, правилами, пользователями, заданиями или фоновой обработкой. Такое разделение снижает связанность системы и упрощает дальнейшее развитие.

Взаимодействие компонентов основано на заранее определенных контрактах данных. Они описывают структуру запросов, ответов и событий, что делает интеграцию более предсказуемой. Для внутренних обращений используется общий клиентский слой, скрывающий детали сетевого взаимодействия.

Файловое содержимое и служебная информация хранятся раздельно. Файлы размещаются в объектном хранилище, а метаданные, сведения о пользователях, правилах и заданиях сохраняются в реляционной базе данных. В рамках MVP используется единая база данных, что упрощает локальный запуск и проверку системы.

Обработка файлов вынесена в асинхронный контур. Создание задания и его выполнение разделены: сначала фиксируется запрос на преобразование, затем отдельный обработчик выполняет его в фоне и возвращает результат. Это позволяет не удерживать пользовательский запрос во время длительной операции и создает основу для масштабирования обработки.

Ключевым архитектурным решением стало использование промежуточного древовидного представления данных. Оно позволяет отделить правила преобразования от конкретного формата входного файла. За счет этого новые форматы и новые правила могут добавляться без изменения общей идеи работы системы.

Для повторно используемой логики выделены отдельные программные модули. В них вынесены операции преобразования форматов, применения правил, доступа к хранилищу и описания общих моделей данных. Это уменьшает дублирование и повышает расширяемость решения.

Дополнительно предусмотрен базовый контур аутентификации, контейнерная инфраструктура для локального запуска и возможность документирования API. В результате архитектура MVP остается достаточно простой для разработки, но сохраняет свойства, необходимые для дальнейшего промышленного развития.

## 2.3. Результаты исследовательского этапа

По итогам исследовательского этапа была определена общая логика работы системы, включающая загрузку файла, настройку правил, создание задания, асинхронную обработку и получение результата. Также был выбран подход с промежуточным древовидным представлением данных, позволяющий отделить правила от конкретных файловых форматов.

Были зафиксированы основные архитектурные принципы MVP: разделение ответственности между компонентами, раздельное хранение файлов и метаданных, использование фоновой обработки для длительных операций, наличие общих контрактов и выделение переиспользуемой логики. Эти решения сформировали техническую основу, достаточную для перехода к реализации.
