# 3. Управление разработкой и координация

Роль участника проекта в процессе разработки определялась как Teamlead и backend developer. Это предполагало совмещение задач организационного управления, технического планирования и непосредственной реализации серверной части системы. Основная цель управления заключалась в том, чтобы в ограниченный срок сформировать работоспособный MVP, покрывающий ключевой пользовательский сценарий и пригодный для дальнейшего расширения.

На этапе планирования была выполнена декомпозиция проекта на функциональные области. Были выделены задачи, связанные с хранением файлов, настройкой правил преобразования, управлением пользователями, созданием заданий, фоновой обработкой и получением результата. Такое разделение позволило определить границы ответственности компонентов и сформировать последовательность разработки, при которой каждый следующий этап опирался на уже реализованную основу.

Особое внимание уделялось определению минимально необходимого состава MVP. В условиях ограниченного срока было важно отделить обязательные функции от возможностей, которые могут быть перенесены на последующие этапы. В качестве обязательных были выделены загрузка файлов, хранение метаданных, настройка правил, запуск задания на преобразование, асинхронная обработка, получение статуса и скачивание результата. Такой подход позволил сохранить фокус на работоспособном сквозном сценарии.

Очередность разработки была выстроена с учетом технических зависимостей. Сначала были определены общие контракты данных и базовые принципы взаимодействия между частями системы. Затем последовательно реализовывались компоненты, отвечающие за самостоятельные функциональные области. После этого выполнялась их интеграция в единый процесс обработки файла. Такой порядок позволил постепенно снижать неопределенность и выявлять ошибки на ранних этапах.

Управление разработкой включало постоянную фиксацию проектных решений в документации. В документации отражались назначение компонентов, ожидаемые сценарии использования, форматы данных, правила преобразования и порядок дальнейшей реализации. Это обеспечивало единое понимание системы и позволяло использовать документацию как рабочий инструмент, а не только как итоговое описание.

В процессе разработки применялся инкрементальный подход. Каждый функциональный блок сначала реализовывался в минимально необходимом объеме, затем проверялся отдельно, после чего подключался к общему сценарию. При возникновении ошибок интеграции выполнялся анализ причины, уточнение конфигурации и внесение точечных исправлений. Такой подход позволил контролировать сложность проекта и не допускать накопления большого количества невыявленных проблем.

Координация также включала подготовку системы к взаимодействию с пользовательским интерфейсом. API проектировались с учетом того, что они должны использоваться не только внутренними компонентами, но и внешним клиентом. Поэтому были предусмотрены операции, необходимые для ручной загрузки файлов, просмотра доступных данных, настройки правил и отслеживания выполнения заданий. Дополнительно были подготовлены материалы, которые позволяют frontend-разработчику понять формат правил и возможности backend-части.

В рамках роли backend developer выполнялась реализация серверной логики, настройка хранения данных, организация асинхронного обмена сообщениями, разработка API и проверка работоспособности компонентов. При этом технические решения принимались с учетом задач MVP: система должна была оставаться достаточно простой для локального запуска, но не должна была блокировать дальнейшее развитие архитектуры.

Результатом управленческой и технической координации стало формирование связанного backend-контура. Были определены границы MVP, реализован основной сценарий обработки, подготовлена инфраструктура для локальной проверки и зафиксированы направления дальнейшего развития. Такой подход позволил совместить скорость реализации с сохранением архитектурной основы, пригодной для последующей доработки.
