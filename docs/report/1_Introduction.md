# 1. Введение

Проект посвящен разработке MVP сервиса файловой конвертации с дополнительной настройкой правил преобразования данных. Актуальность такого решения определяется тем, что в прикладных системах часто требуется передавать данные между компонентами, которые используют разные форматы представления. В рамках проекта рассматриваются структурированные файлы, которые могут быть приведены к внутреннему древовидному представлению, изменены по заданным правилам и сохранены в целевом формате.

Целью разработки являлось создание backend-основы сервиса, которая обеспечивает полный цикл обработки файла: загрузку исходного файла, хранение его содержимого и метаданных, настройку правил преобразования, создание задания на конвертацию, асинхронную обработку задания и получение результата. Для MVP было важно не только реализовать отдельные операции, но и связать их в единый сценарий, пригодный для дальнейшего подключения пользовательского интерфейса и расширения набора поддерживаемых форматов.

В качестве основных задач проекта были выделены проектирование архитектуры, определение контрактов между сервисами, реализация переиспользуемых библиотек, создание backend-сервисов и настройка локальной инфраструктуры. В рамках MVP были реализованы сервисы для работы с файлами, пользователями, правилами преобразования, фасадный сервис для создания заданий и отдельный обработчик конвертации. Также были разработаны клиентские библиотеки для межсервисного взаимодействия и общие DTO-контракты.

Ключевой особенностью проекта является разделение бизнес-процесса на несколько независимых компонентов. Файлы хранятся отдельно от метаданных, правила преобразования конфигурируются через отдельный сервис, а сама конвертация выполняется асинхронно через Kafka. Такой подход позволяет не блокировать пользовательский запрос на время обработки файла и создает основу для масштабирования обработчиков конвертации.

В проекте использован подход, при котором формат входного файла сначала приводится к общему древовидному виду. Далее к этому дереву могут применяться правила, после чего результат сериализуется в целевой формат. Благодаря этому правила преобразования не привязаны напрямую к конкретному формату файла и могут использоваться для разных сценариев обработки.

Роль участника проекта определялась как Teamlead и backend developer. В рамках этой роли выполнялось планирование состава MVP, определение очередности реализации компонентов, формирование документации для разработки, разработка backend-сервисов и библиотек, а также устранение интеграционных ошибок, возникавших при запуске и связывании сервисов. Основное внимание было уделено достижению работоспособного backend-контура, который может быть использован как основа для дальнейшего развития продукта.
