# 3. Управление разработкой и координация frontend-части

Разработка frontend-части выполнялась в рамках общего процесса создания MVP системы файловой конвертации. Основной задачей координации было обеспечить соответствие пользовательского интерфейса backend-логике и требованиям технического задания. Интерфейс должен был не просто отображать отдельные формы, а поддерживать полный пользовательский сценарий от загрузки файла до получения результата.

На этапе планирования были выделены основные пользовательские действия: загрузка файла, выбор целевого формата, просмотр структуры данных, настройка правил преобразования, запуск задания и скачивание результата. Эти действия были сопоставлены с backend API, после чего была определена структура экранов frontend-приложения.

Особое внимание уделялось согласованию frontend и backend моделей данных. Для корректной работы интерфейса требовалось учитывать идентификаторы файлов, целевые форматы, структуру дерева полей, наборы правил, статусы заданий и ссылки на скачивание результата. Такое согласование позволило связать визуальные действия пользователя с фактическими backend-операциями.

Разработка интерфейса велась инкрементально. Сначала была сформирована общая структура приложения и маршрутизация между экранами. Затем были реализованы основные визуальные компоненты: главная страница, экран загрузки, компонент выбора файла, экран структуры и компонент дерева. После этого была выполнена интеграция с backend API, что позволило заменить прототипные данные фактическими ответами сервера.

В рамках координации также учитывались требования к демонстрации проекта. Интерфейс должен позволять наглядно показать комиссии основные возможности системы: загрузку файла, анализ структуры, настройку правил, запуск конвертации и получение результата. Поэтому при проектировании экранов приоритет отдавался понятности сценария и визуальной прозрачности действий.

Результатом координации стала frontend-часть, которая соответствует архитектуре backend и дополняет ее пользовательским уровнем. Пользователь взаимодействует с системой через единый интерфейс, а технические операции выполняются через API. Такой подход делает MVP пригодным для демонстрации и дальнейшего развития.
